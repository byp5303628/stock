# 02. 建设通用的配置表

## 1. 背景与目标

### 1.1 业务背景

在金融业务系统中，各类配置（如交易参数、风控阈值、费率规则、业务开关等）分散在不同的数据表中。每新增一种配置就需要新建一张配置表，导致：

- 配置表数量膨胀，维护成本高
- 不同系统的配置管理方式不统一
- 新增配置类型需要开发 DDL 变更，迭代周期长
- 配置查询缺乏统一入口

### 1.2 业务目标

建设一套通用的配置表系统，提供统一的配置存储与查询能力，满足以下目标：

- 所有配置复用同一张物理表，不再为每种配置单独建表
- 每种配置类型视为一个逻辑表，业务上可独立管理
- 配置详情通过 JSON 存储，支持按属性值筛选
- 查询优先走本地缓存，配置更新时自动刷新缓存

## 2. 需求范围

### 2.1 In Scope

- 通用配置表的存储与查询能力
- 按配置类型查询（每种配置类型作为一个逻辑表）
- 按配置详情中的属性值进行筛选
- 配置详情使用 JSON 格式存储
- 本地缓存优先查询 + 缓存更新机制
- 配置表管理前端页面

### 2.2 Out of Scope

- 配置的变更审批流程（如审批流、版本管理）
- 配置的灰度发布与回滚
- 配置变更的订阅通知（给其他系统推送）
- 配置的导入/导出

## 3. 业务规则

### 3.1 配置定义

- 如果一条配置记录包含配置类型、配置名称、配置详情（JSON），则该记录定义为一个配置条目
- 如果同一配置类型下存在多条配置，则每条配置通过配置名称区分

### 3.2 配置查询

- 如果用户按配置类型查询，则返回该类型下的所有配置条目
- 如果用户在查询时需要按配置详情中的某个属性值过滤，则支持对 JSON 字段中的属性进行筛选
- 如果配置查询命中本地缓存，则直接返回缓存结果，不再访问数据库

### 3.3 缓存策略

- 如果配置数据在数据库中被新增、修改或删除，则对应配置类型的本地缓存必须失效并重新加载
- 如果本地缓存中不存在请求的配置，则回源查询数据库，并将结果写入缓存

## 4. 验收条件

### 4.1 后端

- Given 系统中已存在类型为 "FEE_RULE" 和 "TRADE_PARAM" 两种配置，When 用户按类型查询 "FEE_RULE"，Then 系统返回所有费用规则配置
- Given 配置详情中某条记录的属性 `{ "status": "active" }`，When 用户按 `status = active` 筛选，Then 系统返回所有 status 为 active 的配置
- Given 配置数据已写入数据库，When 用户首次查询某类型配置，Then 系统从数据库读取并写入缓存
- Given 缓存中已有某类型配置，When 用户再次查询同一类型，Then 系统直接从缓存返回，不访问数据库
- Given 数据库中某条配置被更新，When 用户发起查询，Then 系统返回最新数据（缓存已失效并刷新）

### 4.2 前端

- Given 用户打开配置管理页面，When 页面加载完成，Then 展示所有配置类型的列表
- Given 用户在配置类型列表中点击某一类型，When 页面跳转，Then 展示该类型下的所有配置条目列表
- Given 不同配置条目的 JSON 详情字段结构不同，When 用户查看配置条目列表，Then 表格列根据 JSON 属性动态渲染展示

## 5. 非功能性要求

| 要求类别 | 指标 | 目标值 |
|----------|------|--------|
| 查询性能 | 缓存命中时查询响应时间 | < 50ms |
| 查询性能 | 缓存未命中回源查询响应时间 | < 200ms |
| 缓存更新 | 配置更新后缓存失效延迟 | < 5s |
| 可扩展性 | 新增配置类型是否需要改表结构 | 不需要 |
