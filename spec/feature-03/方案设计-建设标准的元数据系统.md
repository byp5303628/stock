# 建设标准的元数据系统 — 方案设计

## 1. 需求概述

### 1.1 业务背景与目标

指标体系（feature-01）和通用配置表（feature-02）需要统一的语义管理能力。当前指标字段和配置属性的业务含义缺乏结构化定义，导致口径理解不一致，AI 模型也无法获取字段语义。

**业务目标：**
- 为指标和配置字段提供可视化语义定义能力
- 枚举系统可绑定到元数据模型属性
- 支持 Schema 展示与校验
- 为本地系统和 AI 大模型提供元数据查询接口

### 1.2 需求范围

| In Scope | Out of Scope |
|----------|-------------|
| 元数据模型可视化配置 | 元数据自动发现与逆向解析 |
| 字段结构化定义 | 版本管理与变更审批流 |
| 枚举系统 | 与其他系统自动同步 |
| Schema 展示与校验 | |
| 元数据 API（本地系统/AI 模型） | |

### 1.3 关键约束

- 元数据模型查询响应 < 200ms
- Schema 校验响应 < 1s
- 支持指标 + 配置表元数据，并支持未来扩展

## 2. 模型设计

### 2.1 领域模型

**核心实体：**

```
MetadataModel（元数据模型）
├── id: Long
├── name: String           // 模型名称，如"市盈率指标"
├── code: String           // 模型编码，如"pe_metric"
├── model_type: String     // 模型类型：INDICATOR / CONFIG
├── description: String    // 业务说明
├── status: String         // DRAFT / PUBLISHED / DEPRECATED
└── ext_info: JSON

MetadataField（元数据字段）
├── id: Long
├── model_id: Long         // 所属模型
├── field_name: String     // 字段名
├── field_type: String     // 字段类型：STRING / NUMBER / BOOLEAN / ENUM / DATE
├── business_meaning: String // 业务含义描述
├── required: Boolean      // 是否必填
├── constraints: JSON      // 约束条件（长度、范围、正则等）
├── enum_id: Long          // 绑定的枚举 ID（可选）
├── sort_order: Integer    // 排序
└── ext_info: JSON

MetadataEnum（枚举定义）
├── id: Long
├── name: String           // 枚举名称，如"财报周期"
├── code: String           // 枚举编码
├── description: String
└── status: String

MetadataEnumValue（枚举值）
├── id: Long
├── enum_id: Long          // 所属枚举
├── value_code: String     // 取值编码
├── value_label: String    // 取值展示名
├── sort_order: Integer
└── ext_info: JSON
```

**实体关系：**

```
MetadataModel 1 ── * MetadataField
MetadataEnum 1 ── * MetadataEnumValue
MetadataEnum 1 ── * MetadataField（按 enum_id 反向引用）
```

> ⚠️ **枚举与多模型绑定关系**：一个枚举可以被多个模型的多个字段引用。例如"财报周期"枚举可能同时被"市盈率指标"模型的 `period` 字段和"营收指标"模型的 `report_period` 字段引用。这意味着枚举的变更会影响多个模型，需要做变更影响分析。

### 2.2 数据模型

#### 新建表：metadata_model

```sql
create table if not exists metadata_model
(
    id          bigint                 not null primary key auto_increment,
    name        varchar(128)           not null comment '模型名称',
    code        varchar(64)            not null comment '模型编码',
    model_type  varchar(32)            not null comment '模型类型',
    description varchar(1024)          not null default '' comment '业务说明',
    status      varchar(16)            not null default 'DRAFT' comment '状态',
    ext_info    json                            comment '扩展属性',
    gmt_create  datetime               not null default now(),
    gmt_modified datetime              not null default now() on update now(),
    unique index uniq_idx_code (code),
    index idx_model_type (model_type),
    index idx_gmt_modified (gmt_modified)
) engine = InnoDB default charset = utf8mb4;
```

#### 新建表：metadata_field

```sql
create table if not exists metadata_field
(
    id              bigint                 not null primary key auto_increment,
    model_id        bigint                 not null comment '所属模型',
    field_name      varchar(64)            not null comment '字段名',
    field_type      varchar(32)            not null comment '字段类型',
    business_meaning varchar(1024)         not null default '' comment '业务含义',
    required        tinyint(1)             not null default 0 comment '是否必填',
    constraints     json                            comment '约束条件',
    enum_id         bigint                          comment '绑定的枚举 ID',
    sort_order      int                    not null default 0 comment '排序',
    ext_info        json                            comment '扩展属性',
    gmt_create      datetime               not null default now(),
    gmt_modified    datetime               not null default now() on update now(),
    index idx_model_id (model_id),
    index idx_enum_id (enum_id),
    index idx_gmt_modified (gmt_modified)
) engine = InnoDB default charset = utf8mb4;
```

#### 新建表：metadata_enum

```sql
create table if not exists metadata_enum
(
    id          bigint                 not null primary key auto_increment,
    name        varchar(128)           not null comment '枚举名称',
    code        varchar(64)            not null comment '枚举编码',
    description varchar(512)           not null default '' comment '描述',
    status      varchar(16)            not null default 'ENABLED' comment '状态',
    gmt_create  datetime               not null default now(),
    gmt_modified datetime              not null default now() on update now(),
    unique index uniq_idx_code (code),
    index idx_gmt_modified (gmt_modified)
) engine = InnoDB default charset = utf8mb4;
```

#### 新建表：metadata_enum_value

```sql
create table if not exists metadata_enum_value
(
    id          bigint                 not null primary key auto_increment,
    enum_id     bigint                 not null comment '所属枚举',
    value_code  varchar(64)            not null comment '取值编码',
    value_label varchar(128)           not null comment '取值展示名',
    sort_order  int                    not null default 0 comment '排序',
    ext_info    json                            comment '扩展属性',
    gmt_create  datetime               not null default now(),
    gmt_modified datetime              not null default now() on update now(),
    index idx_enum_id (enum_id),
    index idx_gmt_modified (gmt_modified)
) engine = InnoDB default charset = utf8mb4;
```

> `metadata_field.enum_id` 上已建索引 `idx_enum_id`，用于高效反向查询"某个枚举被哪些字段引用"——这是枚举在多模型间共享时的核心查询。

### 2.3 枚举与多模型绑定的设计考量

#### 2.3.1 当前设计支持的场景

```
枚举（财报周期）
├── 枚举值：Q1, Q2, Q3, Q4
│
├── 被模型 A（市盈率指标）的 period 字段引用   ← model_A.field_1.enum_id
└── 被模型 B（营收指标）的 report_period 字段引用 ← model_B.field_2.enum_id
```

从**模型→枚举**方向：`MetadataField.enum_id → MetadataEnum.id`，当前设计已天然支持多模型引用。

从**枚举→模型**方向：通过 `SELECT * FROM metadata_field WHERE enum_id = ?` 可查询枚举被哪些字段引用、这些字段属于哪些模型。

#### 2.3.2 变更影响分析

| 枚举变更操作 | 影响范围 | 应对策略 |
|-------------|---------|---------|
| **新增枚举值** | 无影响 | 新增值对所有引用方立即可用 |
| **修改枚举值 label** | 展示层影响 | 所有引用字段的展示名同步变化，属于预期行为 |
| **修改枚举值 code** | ⚠️ 破坏性变更 | 禁止直接修改；如有需要先确认无引用，或通过新建枚举的方式迁移 |
| **删除枚举值** | ⚠️ 破坏性变更 | 检查是否有已有数据引用该值，有引用时禁止删除 |
| **删除枚举** | ⚠️ 破坏性变更 | 检查 `metadata_field` 中是否有字段引用，有则提示"被 N 个字段引用，请先解除绑定" |
| **修改枚举名称** | 展示层影响 | 在所有引用处同步更新，属于预期行为 |

#### 2.3.3 软删除策略

枚举和枚举值采用软删除：

- `metadata_enum` 增加 `status` 字段标记删除（`ENABLED / DISABLED / DELETED`）
- 引用该枚举的字段在查询时自动过滤 `status != DELETED` 的枚举
- 已删除的枚举不影响已有历史数据的解析

#### 2.3.4 Schema 校验补充规则

在原有校验逻辑基础上，增加枚举绑定校验：

```java
// 枚举绑定校验（全模型发布时执行）
for (每个引用了枚举的字段) {
    if (enum.status == DELETED) → 错误："字段 xxx 引用的枚举已被删除"
    if (枚举值列表为空) → 警告："枚举 xxx 没有定义任何枚举值"
    if (约束条件中指定了枚举值但该值不存在) → 错误
}
```

#### 2.3.5 前端交互

| 页面 | 交互要点 |
|------|---------|
| **枚举列表页** | 每行展示引用数（如"被 2 个模型、3 个字段引用"），点击可查看详情 |
| **枚举详情页** | 展示"引用此枚举的字段"列表（字段名 → 所属模型 → 模型类型），支持跳转到对应模型 |
| **枚举编辑页** | 如果已被引用，显示提示"⚠️ 该枚举被 N 个字段引用，修改将影响这些模型" |
| **枚举删除** | 有引用时禁止删除，提示"请先解除所有绑定" |
| **模型字段编辑** | 选择枚举时展示引用热度（"此枚举还被其他 3 个字段使用"） |

### 2.4 缓存模型

元数据模型的变更频率极低（创建后基本不变），适合全量缓存：

| 数据 | 策略 | TTL |
|------|------|-----|
| 元数据模型 | 全量加载到本地 Caffeine 缓存 | 30 分钟 |
| 枚举定义 | 全量加载到本地 Caffeine 缓存 | 30 分钟 |
| Schema 校验 | 不缓存，实时执行 | — |

写入/更新元数据时，通过缓存失效事件使本地缓存失效，下次查询重新加载。

## 3. 架构设计

### 3.1 模块划分

| Maven 模块 | 新增/修改 | 职责 |
|-----------|----------|------|
| `stock-common` | 新增 | Mapper、DO、XML（4 张表） |
| `stock-core` | 新增 | MetadataDomainService、领域模型 |
| `stock-biz` | 新增 | MetadataController、枚举管理 |

前端（stock-frontend）需新增：
- 元数据模型列表页 + 编辑页
- 枚举管理页
- 模型详情展示页（含 Schema 展示）

### 3.2 接口设计

**元数据模型：**
```
GET    /api/metadata/model/list.json          — 模型列表
GET    /api/metadata/model/detail.json?id=    — 模型详情（含字段）
POST   /api/metadata/model/save.json          — 新建/更新模型
POST   /api/metadata/model/validate.json      — Schema 校验
```

**字段管理：**
```
POST   /api/metadata/field/save.json          — 新增/更新字段
DELETE /api/metadata/field/delete.json?id=    — 删除字段
```

**枚举管理：**
```
GET    /api/metadata/enum/list.json           — 枚举列表（含各枚举的引用数）
POST   /api/metadata/enum/save.json           — 新建/更新枚举
GET    /api/metadata/enum/detail.json?id=     — 枚举详情（含枚举值 + 引用字段列表）
POST   /api/metadata/enum/bind.json           — 绑定枚举到字段
POST   /api/metadata/enum/unbind.json         — 解除枚举绑定
GET    /api/metadata/enum/usage.json?id=      — 查询枚举被哪些模型/字段引用（含引用热度）

**本地集成：**
```
GET    /api/metadata/indicator/meaning?code=  — 查询指标含义（供本地系统）
GET    /api/metadata/indicator/usage?code=    — 查询指标用法（供 AI 模型）
```

### 3.3 Schema 校验逻辑

```java
validateSchema(modelId) {
    1. 加载该模型的所有字段定义
    2. 对每个字段校验：
       - 字段名是否为空
       - 字段类型是否在合法列表中
       - 如果 field_type=ENUM，enum_id 是否必填且存在
       - constraints 格式是否合法
       - required 字段至少要有 business_meaning
    3. 校验通过 → 更新模型状态为 PUBLISHED
    4. 校验不通过 → 返回错误明细列表
}
```

## 4. 影响面分析

- **新增表 4 张**：不涉及现有表变更
- **前端新增**：元数据管理页面不影响现有页面
- **与 feature-01/02 的关联**：元数据系统独立部署，通过 API 被指标系统和配置表系统调用，无直接依赖
