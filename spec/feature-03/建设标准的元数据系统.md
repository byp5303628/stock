# 03. 建设标准的元数据系统

## 1. 背景与目标

### 1.1 业务背景

在指标体系（feature-01）和通用配置表（feature-02）的建设过程中，指标字段和配置字段缺乏统一的语义描述和管理机制。当前存在以下问题：

- 指标的各个字段（如 PE 的分子/分母口径）缺乏结构化定义，业务含义仅存在于文档中
- 配置表中 JSON 字段的属性含义不透明，使用者无法直观理解每个字段的业务语义
- 不同系统对同一字段的理解不一致，导致指标口径出现偏差
- AI 模型在读取指标时无法获取指标的业务含义和使用逻辑

### 1.2 业务目标

建设一套标准的元数据管理系统，为指标和配置的字段提供语义定义能力，满足以下目标：

- 通过可视化页面配置，管理指标和配置中各字段的结构与语义
- 支持枚举系统，可绑定到元数据模型及其属性上
- 支持元数据模型的展示、Schema 展示及校验
- 为本地系统提供指标含义查询能力
- 为本地 AI 大模型提供指标用法和数据取数逻辑

## 2. 需求范围

### 2.1 In Scope

- 元数据模型的可视化配置与管理页面
- 元数据字段的结构化定义（字段名、类型、语义描述、约束）
- 枚举系统的定义与管理（可绑定到元数据模型及属性）
- 元数据模型的 Schema 展示与校验
- 元数据系统的 API 接口，供本地系统查询指标含义
- 供 AI 大模型使用的指标用法和取数逻辑接口

### 2.2 Out of Scope

- 元数据的自动发现与逆向解析（如从数据库表结构自动生成元数据）
- 元数据的版本管理与变更审批流
- 元数据与其他系统（如数据地图）的自动同步

## 3. 业务规则

### 3.1 元数据模型

- 如果一个模型定义了某指标的字段结构，则该模型包含字段名称、字段类型、业务含义、约束条件四项基本属性
- 如果一个字段的业务含义可以被枚举值限定，则将该字段绑定到对应的枚举系统
- 如果元数据的 Schema 发生变更，则需要对变更后的 Schema 进行校验，校验通过方可生效

### 3.2 枚举系统

- 如果元数据模型中的某个属性的取值范围是固定的，则定义一个枚举类型并绑定到该属性上
- 如果一个枚举类型被多个元数据属性引用，则枚举值变更时所有引用方同步生效

### 3.3 本地使用

- 如果本地系统需要查询某指标的含义，则通过元数据 API 返回该指标的完整字段定义和语义描述
- 如果 AI 大模型需要理解某指标的用法，则通过元数据 API 返回该指标的业务逻辑说明和数据取数规则

## 4. 验收条件

### 4.1 元数据管理

- Given 用户进入元数据管理页面，When 用户新建一个元数据模型并定义字段结构，Then 模型创建成功并可查看
- Given 元数据模型中某个字段需要限定取值范围，When 用户创建一个枚举类型并绑定到该字段，Then 该字段的取值被枚举约束
- Given 用户修改了元数据模型的 Schema，When 修改内容不符合校验规则，Then 系统提示错误并阻止生效

### 4.2 模型展示

- Given 系统中已存在一个元数据模型，When 用户查看模型详情，Then 展示该模型的完整 Schema 和字段定义

### 4.3 本地集成

- Given 本地系统调用元数据 API 查询某指标含义，When 请求合法，Then 返回该指标的所有字段定义和语义描述
- Given AI 大模型调用元数据 API 获取指标用法，When 请求合法，Then 返回该指标的业务逻辑说明和取数规则

## 5. 非功能性要求

| 要求类别 | 指标 | 目标值 |
|----------|------|--------|
| 查询性能 | 元数据模型查询响应时间 | < 200ms |
| 查询性能 | Schema 校验响应时间 | < 1s |
| 兼容性 | 支持绑定的元数据模型类型 | 指标 + 配置表 + 后续可扩展 |
