# 建设通用的指标体系 — 方案设计（v2）

## 1. 需求概述

### 1.1 业务背景与目标

投研人员需要统一的指标存储与查询能力，解决当前指标分散、口径不一致、查询不便、新增指标迭代周期长的问题。

**业务目标：**
- 统一的金融指标存储与查询
- 多维度查询（行业/股票/时间）
- 指标灵活扩充（无需改表结构）
- 本地查询与当前值获取

### 1.2 需求范围

| In Scope | Out of Scope |
|----------|-------------|
| 通用指标存储与查询 | 数据源采集与同步 |
| 行业维度指标查询 | 可视化展示 |
| 个股历史指标查询 | 指标计算与加工 |
| 指标灵活扩充 | |

### 1.3 关键约束

- 新增指标不得调整数据库表结构
- 单指标查询响应 < 500ms
- 行业聚合查询响应 < 3s
- 支持不低于 1000 种指标

## 2. 模型设计

### 2.1 领域模型

**核心实体：**

```
IndicatorDefinition（指标定义）
├── id: Long
├── name: String          // 指标名称
├── code: String          // 指标编码
├── category: String      // 指标分类
├── unit: String          // 单位
├── description: String
├── ext_info: JSON
└── status: String        // ENABLED / DISABLED

IndicatorValue（指标值）
├── id: Long
├── indicator_code: String
├── entity_type: String   // STOCK / INDUSTRY
├── entity_code: String
├── partition_date: String
├── value: BigDecimal
├── ext_info: JSON
└── gmt_create/modified
```

### 2.2 数据模型

#### 新建表：indicator_definition

```sql
create table if not exists indicator_definition
(
    id          bigint                 not null primary key auto_increment,
    code        varchar(64)            not null comment '指标编码',
    name        varchar(128)           not null comment '指标名称',
    category    varchar(32)            not null comment '指标分类',
    unit        varchar(32)            not null default '' comment '单位',
    description varchar(512)           not null default '' comment '指标说明',
    ext_info    json                            comment '扩展属性',
    status      varchar(16)            not null default 'ENABLED' comment '状态',
    gmt_create  datetime               not null default now(),
    gmt_modified datetime              not null default now() on update now(),
    unique index uniq_idx_code (code),
    index idx_category (category),
    index idx_gmt_modified (gmt_modified)
) engine = InnoDB default charset = utf8mb4;
```

#### 新建分区表：indicator_value

```sql
create table if not exists indicator_value
(
    id              bigint                 not null primary key auto_increment,
    indicator_code  varchar(64)            not null comment '指标编码',
    entity_type     varchar(16)            not null comment '实体类型: STOCK/INDUSTRY',
    entity_code     varchar(32)            not null comment '实体代码',
    partition_date  varchar(32)            not null comment '日期',
    value           decimal(30, 6)         not null comment '指标值',
    ext_info        json                            comment '扩展属性',
    gmt_create      datetime               not null default now(),
    gmt_modified    datetime               not null default now() on update now(),
    unique index uniq_idx_indicator_entity_date (indicator_code, entity_type, entity_code, partition_date),
    index idx_entity_type_code_date (entity_type, entity_code, partition_date),
    index idx_gmt_modified (gmt_modified)
) engine = InnoDB default charset = utf8mb4
partition by range columns (partition_date)
(
    partition p_2024q1 values less than ('2024-04-01'),
    partition p_2024q2 values less than ('2024-07-01'),
    partition p_2024q3 values less than ('2024-10-01'),
    partition p_2024q4 values less than ('2025-01-01'),
    partition p_2025q1 values less than ('2025-04-01'),
    partition p_future values less than maxvalue
);
```

> **数据量评估**：A 股约 5000 只股票 × 200 种指标 × 日均数据，按季度分区控制单分区数据量在 1 亿行以内。`indicator_code` 作为查询第一过滤条件，配合唯一索引 `uniq_idx_indicator_entity_date` 可高效定位。超过 50 亿行后可引入 TiDB 或 ClickHouse 做存储层。

### 2.3 缓存模型

| 层级 | 方案 | TTL | 说明 |
|------|------|-----|------|
| L1 | Caffeine 本地缓存 | 10 分钟 | 指标定义信息（极少变化） |
| L2 | Redis | 5 分钟 | 热点指标值 |
| DB | MySQL 直查 | — | 兜底 |

> **非功能保障**：Redis 不可用时自动降级到 DB 查询，降级期间增加本地缓存 TTL 缓解 DB 压力。缓存击穿通过互斥锁（`synchronized` + 双重检查）保护。

## 3. 架构设计

### 3.1 模块划分

## 4. 评论响应

根据文档评论中的反馈，以下是对应的处理：

| 评论 | 处理 |
|------|------|
| 一张表是否能支撑大量数据（5000 只 × 200 指标 × 多时间节点） | 已增加分区表设计 + 分区剪枝说明，预留 OLAP 扩展点 |
| 非功能性问题如何保障 | 已补充缓存降级策略 + 并发保护 + 分区策略 |
