# Rich — PRD 润色命令

接收一个飞书 PRD 文档或一段需求描述，对内容进行业务语义润色，消除二义性。润色完成后本地保存到 `spec/` 目录。

- 如果传入的是**飞书文档链接**：读取文档 + 评论 -> 润色 -> 本地保存 -> 写回飞书
- 如果传入的是**需求描述文本**：在本地 `spec/` 创建需求目录 -> 润色 -> 本地保存（不上传飞书）

## 参数

```
/rich <飞书文档链接|需求描述>
```

- `<飞书文档链接>` — 飞书云文档或知识库文档的 URL
- `<需求描述>` — 纯文本的需求描述（如"给 stock-common 增加单测覆盖"）

## 执行流程

### Step 0: 判断输入类型

检查参数是否为飞书 URL（包含 `feishu.cn` / `lark-` 等特征）：

```bash
if [[ "$1" =~ (feishu\.cn|feishu|lark-) ]]; then
  MODE="feishu"
else
  MODE="requirement"
fi
```

- **飞书文档模式**：走 Step 1a -> Step 2 -> Step 3 -> Step 4
- **需求描述模式**：走 Step 1b -> Step 2 -> Step 3（跳过 Step 4）

### Step 1a: 飞书文档模式

当输入为飞书链接时：

1. 调用 **lark-doc** skill 读取文档全文内容
2. 调用 **lark-drive** skill 读取该文档的评论列表，将评论内容及对应的文档段落位置一并提取
3. 整合文档正文 + 评论，形成完整输入素材
4. 从文档标题推断需求名称和序号：
   - 如果文档标题包含 `{序号}-{名称}` 格式则直接解析
   - 否则以文档标题作为需求名称，序号查看 `spec/` 目录下已有编号自增

### Step 1b: 需求描述模式

当输入为纯文本需求描述时：

1. 直接将输入的需求描述作为输入素材
2. 从需求描述中提取关键词作为需求名称（如"给 stock-common 增加单测覆盖" -> `add-unit-tests`）
3. 查看 `spec/` 目录下已有编号，取下一个序号

### Step 2: PRD 润色（融合评论反馈）

对输入素材进行业务语义润色，同时将 **评论中的反馈、疑问、建议** 纳入处理（仅飞书模式有评论）：

| 原则 | 说明 |
|------|------|
| **消除二义性** | 每个需求点、验收条件只能有一种理解方式，禁止"大概""可能""酌情"等模糊表述 |
| **业务语义清晰** | 从业务方视角描述，说明"谁在什么场景下要做什么"，而非技术实现 |
| **无技术偏好** | 不预设技术选型，只描述业务需求本身 |
| **结构化** | 按"背景 -> 目标 -> 需求范围 -> 业务规则 -> 验收条件 -> 非功能性要求"组织 |
| **用语统一** | 全文保持术语一致，不混用同义表述 |

润色覆盖以下各节（需求描述模式从文本中推断结构）：
- **背景与目标** — 明确业务动机和衡量指标
- **需求范围** — 区分 In Scope / Out of Scope，边界清晰
- **业务规则** — 每一条规则用 "如果...则..." 句式，无歧义
- **验收条件** — 每个条件可独立验收，表述为 Given-When-Then 格式
- **非功能性要求** — 明确量级（如"页面加载 < 2s"，而非"性能要好"）

#### 评论处理规则（仅飞书模式）

评论中的内容应作为润色参考，处理方式如下：

| 评论类型 | 处理方式 |
|----------|----------|
| **疑问类** | 在对应段落补充清晰定义，消除歧义 |
| **建议类** | 评估后补充到对应章节，并在变更摘要中标注 |
| **确认类** | 将确认结果固化到正文，避免后续反复确认 |
| **反对/争议类** | 保持争议内容在评论中不做修改，在变更摘要中标记"待决策" |
| **无实质内容** | 忽略，不影响正文 |

> 原则：评论反映了评审方对 PRD 的关切点，润色时吸收合理反馈使文档更完善，但不越权做决策。如果评论较多，在润色完成后输出评论处理汇总表。

### Step 3: 本地保存

1. 确保 `spec/` 目录存在
2. 以 `spec/feature-{序号}/{需求名称}.md` 格式保存润色后的内容
3. 如果本地已有同名文件，询问用户是否覆盖
4. 记录本次润色的变更摘要

### Step 4: 写回飞书（仅飞书文档模式）

- 仅当输入为飞书文档链接时执行
- 需求描述模式**跳过此步骤**

1. 调用 **lark-doc** skill，将润色后的完整内容写回原飞书文档
2. 确认写入成功后，在回复中附上飞书文档链接和本地路径

## 输出示例

### 飞书文档模式
```
✅ PRD 润色完成！

📄 飞书文档: https://xxx.feishu.cn/docx/xxxxx
📁 本地备份: spec/feature-012/订单逆向流程优化.md

润色变更摘要:
- 消除二义性: 8处 -> 明确为具体条件
- 补充边界: 3处 -> 补充异常场景处理
- 术语统一: "退款" / "退费" -> 统一为"退款"
- 验收条件: 6条 -> 全部转为 Given-When-Then 格式
- 评论处理: 吸收建议 3条, 消除疑问 2条, 待决策 1条
```

### 需求描述模式
```
✅ 需求润色完成！

📁 本地保存: spec/feature-013/add-unit-tests.md

润色变更摘要:
- 明确需求范围: 确定 In Scope / Out of Scope 边界
- 补充验收条件: 覆盖正常流程和异常场景
- 非功能性要求: 明确覆盖率目标
```

## 注意事项

- 飞书文档模式：润色不改变需求的原始意图，只优化表述
- 需求描述模式：从描述文本推断结构，补充缺失章节但不臆造需求
- 需求描述模式不会请求飞书 API，也不需要飞书权限
- 如果文档内容不完整（缺少关键章节），在回复中列出缺失部分建议补充
- 写回飞书前确认文档权限可写
