# Rich — PRD 润色命令

接收一个飞书 PRD 文档链接，对内容进行业务语义润色，消除二义性，使需求描述清晰、业务方向明确。润色完成后，本地保留一份副本并自动写回飞书文档。

## 参数

```
/rich <飞书文档链接>
```

- `<飞书文档链接>` — 飞书云文档或知识库文档的 URL（必填）

## 执行流程

### Step 1: 解析参数并读取飞书文档（含评论）

1. 从命令参数中提取飞书文档 URL / token
2. 调用 **lark-doc** skill 读取文档全文内容
3. 调用 **lark-drive** skill 读取该文档的评论列表（文档评论通常包含评审人对需求的反馩、疑问、建议），将评论内容及对应的文档段落位置一并提取
4. 整合文档正文 + 评论，形成完整的输入素材
5. 从内容中推断需求名称和序号：
   - 如果文档标题包含 `{序号}-{名称}` 格式则直接解析
   - 否则以文档标题作为需求名称，序号查看 `spec/` 目录下已有编号自增

### Step 2: PRD 润色（融合评论反馈）

对文档全文进行业务语义润色，同时将 **评论中的反馈、疑问、建议** 纳入处理：

| 原则 | 说明 |
|------|------|
| **消除二义性** | 每个需求点、验收条件只能有一种理解方式，禁止"大概""可能""酌情"等模糊表述 |
| **业务语义清晰** | 从业务方视角描述，说明"谁在什么场景下要做什么"，而非技术实现 |
| **无技术偏好** | 不预设技术选型，只描述业务需求本身 |
| **结构化** | 按"背景→目标→需求范围→业务规则→验收条件→非功能性要求"组织 |
| **用语统一** | 全文保持术语一致，不混用同义表述 |

润色覆盖以下 PRD 各节：
- **背景与目标** — 明确业务动机和衡量指标
- **需求范围** — 区分 In Scope / Out of Scope，边界清晰
- **业务规则** — 每一条规则用 "如果…则…" 句式，无歧义
- **验收条件** — 每个条件可独立验收，表述为 Given-When-Then 格式
- **非功能性要求** — 明确量级（如"页面加载 < 2s"，而非"性能要好"）

#### 评论处理规则

评论中的内容应作为润色参考，处理方式如下：

| 评论类型 | 处理方式 |
|----------|----------|
| **疑问类**（"这个地方是要做 X 还是 Y？"） | 在对应段落补充清晰定义，消除歧义 |
| **建议类**（"建议加一个 xx 的判断"） | 评估后补充到对应章节，并在变更摘要中标注 |
| **确认类**（"这里确认一下是走 A 流程"） | 将确认结果固化到正文，避免后续反复确认 |
| **反对/争议类** | 保持争议内容在评论中不做修改，在变更摘要中标记"待决策" |
| **无实质内容**（"+1""收到"等） | 忽略，不影响正文 |

> 原则：评论反映了评审方对 PRD 的关切点，润色时吸收合理反馈使文档更完善，但**不越权做决策**——有争议的部分保持现状并在摘要中提示。如果评论较多，在润色完成后输出评论处理汇总表。

### Step 3: 本地保存

1. 确保 `spec/` 目录存在
2. 以 `spec/feature-{序号}/{需求名称}.md` 格式保存润色后的内容
3. 如果本地已有同名文件，询问用户是否覆盖

### Step 4: 写回飞书

1. 调用 **lark-doc** skill，将润色后的完整内容写回原飞书文档
2. 确认写入成功后，在回复中附上飞书文档链接和本地路径

## 输出示例

```
✅ PRD 润色完成！

📄 飞书文档: https://xxx.feishu.cn/docx/xxxxx
📁 本地备份: spec/feature-012/订单逆向流程优化.md

润色变更摘要:
- 消除二义性: 8处 → 明确为具体条件
- 补充边界: 3处 → 补充异常场景处理
- 术语统一: "退款" / "退费" → 统一为"退款"
- 验收条件: 6条 → 全部转为 Given-When-Then 格式
- 评论处理: 吸收建议 3条, 消除疑问 2条, 待决策 1条 ⚠️

📌 评论处理明细:
| 评论位置 | 类型 | 处理结果 |
|----------|------|----------|
| 3.2 退款流程 | 建议 | 已补充超时自动退款逻辑 |
| 4.1 对账范围 | 疑问 | 已明确为"仅含线上支付订单" |
| 5.3 审批节点 | 争议 | ⚠️ 待产品决策 |
```

## 注意事项

- 润色不改变需求的原始意图，只优化表述
- 如果文档内容不完整（缺少关键章节），在回复中列出缺失部分建议补充
- 写回飞书前确认文档权限可写
